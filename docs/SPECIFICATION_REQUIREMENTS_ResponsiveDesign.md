# 要件定義書: レスポンシブデザイン改修（スマホ・iPad 対応）

- 作成日: 2026-07-13
- 作成者: Fable（メインセッション）
- 対象プロダクト: GeOsaka（大阪の場所当てゲーム / React + Vite + TypeScript）
- 関連仕様: docs/SPECIFICATION_DESIGN_GeOsaka.md, docs/SPECIFICATION_*_StreetViewQuestion.md

## 1. 背景と目的

GeOsaka は現在デスクトップ想定のレイアウトで、レスポンシブ対応は `@media(max-width:780px)` の 1 ブレークポイントのみ（src/index.css / src/feature.css）。スマートフォンと iPad で表示が崩れる・使いにくい箇所があるため、両デバイスで「きれいに表示され、快適に操作できる」状態に改修する。

## 2. 対象デバイス・ビューポート

| 区分 | 代表ビューポート | 備考 |
|------|----------------|------|
| スマホ（縦） | 320×568 〜 430×932 | iPhone SE 〜 iPhone Pro Max 相当。最小 320px を下限とする |
| スマホ（横） | 〜932×430 | 高さが極端に低いケースを考慮 |
| iPad（縦） | 768×1024, 810×1080, 834×1194 | **現状 780px 境界の直下/直上で挙動が割れる帯域。要専用設計** |
| iPad（横） | 1024×768, 1194×834 | デスクトップ 2 カラムの縮小版が成立するか要検証 |
| デスクトップ | 1280×800 以上 | 現行デザインを維持（デグレ禁止） |

## 3. 現状の問題点（コード調査に基づく）

1. **ブレークポイントが 780px の 1 段のみ**。iPad 縦（768px）はスマホ扱い、iPad 横（1024px）はデスクトップ扱いになり、どちらも中間帯向けの最適化がない。
2. **`round-grid` が `minmax(330px,.8fr)` の固定最小幅**を持ち、781〜900px 程度の幅でシーン表示部が過度に圧迫される。
3. **`min-height:100vh` / `69vh` などの vh 依存**。モバイルブラウザのアドレスバー伸縮でジャンプ・見切れが起きる（`dvh`/`svh` 未使用）。
4. **セーフエリア（`env(safe-area-inset-*)`）未対応**。ホームインジケータ付き iPhone で sticky の確定ボタン（`.confirm`）が操作しにくい。
5. **タイトル `h1` の `clamp(72px,15vw,210px)` + `letter-spacing:-.11em`** が 320〜375px でオーバーフロー気味。`.title-screen` の透視グリッド背景・orb も小画面で過大。
6. **タップターゲット**: `.legal-links`（11px）、`.howto` など 44px 未満の要素がある。
7. **リザルト画面**: `.result-hero` の距離ブロック、`.reveal` の 3 カラム、`.breakdown` の 4 カラムはスマホで折返し調整済みだが、iPad 帯での検証がない。`.next-button` の `float:right` は狭幅で扱いにくい。
8. **地図・ストリートビューの高さ**が固定的（`.map-canvas` 320px / viewer `min(69vh,720px)`）。スマホ横向きで操作領域が確保できない。
9. **横スクロールの発生有無**が体系的に検証されていない。

## 4. 機能要件

### FR-1 ブレークポイント体系の再設計
- 少なくとも「スマホ（〜767px）/ タブレット（768〜1099px 程度）/ デスクトップ」の 3 段構成にする。境界値は設計フェーズで確定してよいが、iPad 縦 768px・iPad 横 1024px の両方が「タブレット向けに調整されたレイアウト」で表示されること。
- iPad 縦: ラウンド画面は 1 カラム（シーン上・地図下）または成立する 2 カラムのいずれかを設計で選定。iPad 横: 2 カラム維持で比率を調整。

### FR-2 ビューポート単位の近代化
- 画面全体の高さ指定は `100dvh`（フォールバック `100vh`）へ移行。シーンビューア等の `vh` 依存箇所も同様に見直す。

### FR-3 セーフエリア対応
- sticky の確定ボタン、画面下部の余白に `env(safe-area-inset-bottom)` を反映。左右の inset も全画面レイアウトで考慮。

### FR-4 タイトル/最終結果画面の小画面調整
- 320px 幅でも `h1`・コピー・ボタン・HOW TO・legal links がオーバーフローせず収まる。orb・背景グリッドは小画面で縮小または簡略化してよい（見た目の世界観は維持）。

### FR-5 ラウンド画面の操作性
- スマホ縦: シーン → 地図 → 確定ボタンが自然な縦フローで並び、確定ボタンは sticky を維持。地図の高さはピン操作に十分（目安 40vh 前後、設計で確定）。
- スマホ横: シーンと地図が両方視認できるレイアウト（高さ最適化）。
- タブレット: シーン・地図とも十分な面積を確保。

### FR-6 リザルト/最終結果画面の調整
- `.result-hero` `.reveal` `.breakdown` `.next-button` がスマホ・iPad の各帯で崩れなく表示される。`float` 依存を廃止してよい。

### FR-7 タップターゲット
- 主要な操作要素（ボタン、リンク）はタップ領域 44×44px 以上を確保する（見た目のフォントサイズは維持してよい）。

### FR-8 横スクロール禁止
- 全画面・全対象ビューポートで意図しない横スクロールが発生しないこと。

## 5. 非機能要件

- **NFR-1 デグレ禁止**: デスクトップ（1280px 以上）の現行ビジュアルを維持する。
- **NFR-2 挙動不変**: ゲームロジック・コンポーネントの機能的挙動（props/state/イベント）は変更しない。変更は原則 CSS と、レイアウト目的の最小限のマークアップ（className 追加等）に留める。
- **NFR-3 依存追加なし**: 新規ライブラリ・CSS フレームワークは導入しない。
- **NFR-4 パフォーマンス**: ビルドサイズへの影響は軽微であること。

## 6. 非対象（スコープ外）

- ビジュアルデザインの刷新（配色・タイポグラフィの世界観変更）
- PWA 化、オフライン対応
- Google Maps / Street View の機能変更（表示領域のサイズ調整のみ対象）

## 7. 受け入れ基準

1. 次の各ビューポートで全 4 画面（タイトル / ラウンド / ラウンド結果 / 最終結果）に横スクロール・要素の見切れ・重なりがない: 320×568, 375×812, 430×932, 932×430(横), 768×1024, 1024×768, 1280×800。
2. スマホ縦でラウンド画面の「ピンを置く → 確定」操作がスクロール最小限で完結する（確定ボタン常時到達可能）。
3. `npm run build` が成功し、既存テスト（tests/ 配下）が全てパスする。
4. デスクトップ表示のスクリーンショット比較で意図しない変化がない（目視確認で可）。

## 8. 検証方法

- Vite dev サーバー + ブラウザのビューポートリサイズによる目視確認（スクリーンショット取得）。
- `npm run build` / 既存ユニットテストの実行。
- 追加テスト: CSS 中心の改修のためユニットテストの新規追加は必須としないが、レイアウトに関わるロジック（もしあれば）を変更した場合はテストを追加する。
