# 要件定義書: GeOsaka ランダム地点ストリートビュー出題機能

作成日: 2026-07-13 / 作成者: Fable（要件定義担当）

## 1. 背景

GeOsaka は現在、大阪府内の厳選15スポットの写真を見て場所を推測する「写真パネル方式」のゲームである（[SPECIFICATION_REQUIREMENTS_GeOsaka.md](SPECIFICATION_REQUIREMENTS_GeOsaka.md)）。既に結果画面には正解地点を確認できる Google Maps 外部リンク方式のストリートビュー機能が追加済みである（[SPECIFICATION_REQUIREMENTS_StreetView.md](SPECIFICATION_REQUIREMENTS_StreetView.md)）。

本書は、**出題そのもの**を変更する新機能を定義する。ラウンド開始時に提示する「問題」を、厳選スポットの写真ではなく、**大阪府内の完全にランダムな地点のストリートビュー**に置き換える。GeoGuessr に近い体験へゲームの核を変更するものであり、既存の写真パネル方式・スポットデータとは独立した出題経路を新設する。

## 2. ユーザー決定事項（本書の前提）

要件確認のため実施したヒアリングで、以下が確定した。

1. **出題範囲**: 既存15スポットの座標は使わない。大阪府の境界内で毎ラウンド動的に緯度経度を生成し、その地点のストリートビューを問題として表示する。正解は生成した座標そのものであり、スポット名・解説文は存在しない。
2. **APIキー方針**: 既存要件定義書が定める「APIキー・課金登録不要」の原則を、本機能に限り変更する。Google Maps Platform の API キーを導入してよい。ただし、APIキー未設定でもビルド・起動が壊れないフォールバックを用意する。

## 3. 前提・制約の再確認

- 本機能は既存の写真パネル方式を**置き換える**。`RoundScreen` が写真の代わりにストリートビュー・パノラマを表示する。
- 既存の推測フロー（`GuessMap` へのピン設置、確定操作、`gameReducer` の phase 遷移）、採点ロジック（`haversine` 距離、`score.ts`）は変更しない。「問題として何を表示するか」だけを差し替える。
- 既存の結果画面ストリートビュー外部リンク機能（[SPECIFICATION_REQUIREMENTS_StreetView.md](SPECIFICATION_REQUIREMENTS_StreetView.md)）はそのまま維持する。ランダム地点出題時にも、正解座標への外部リンクとして引き続き有効に機能する。
- プロジェクトは Cloudflare Pages の Direct Upload で公開されている（[DEPLOY_CloudflarePagesCLI.md](DEPLOY_CloudflarePagesCLI.md)）。ビルドはローカル実行であり、サーバーサイド・Functions は存在しない。API キーはクライアントに埋め込まれるビルド時環境変数として扱う（Google Maps Platform の Web 向けキーは HTTP リファラー制限で保護する運用とし、シークレット扱いの厳密な秘匿はスコープ外とする）。
- API キー未設定時は、**既存の写真パネル方式（`StaticPhotoSceneProvider` による15スポット出題）にフォールバックする**。アプリがクラッシュしたり、空白画面になったりしてはならない。

## 4. 機能要件

### 4.1 出題地点の生成

- ラウンドごとに、大阪府の地理的範囲内で緯度経度をランダム生成する。
- 生成した地点にストリートビュー画像（Google のパノラマ）が存在するとは限らないため、**メタデータ確認 → 画像なしなら再抽選**という手順を踏む。無限ループを避けるため再試行回数の上限を設ける。
- 上限回数まで試しても有効な地点が見つからない場合、そのラウンドは代替手段（後述 4.4）にフォールバックする。
- 大阪府の範囲判定は、MVP では**矩形（バウンディングボックス）判定で可**とする（府境の複雑なポリゴン判定は求めない）。ただし矩形が明らかに府外の海上・隣接府県を広く含む場合は、ストリートビュー有無の確認による自然な絞り込みで補う。

### 4.2 出題画面の表示内容

- `RoundScreen` の写真表示部分を、生成された地点のストリートビュー・パノラマ（ドラッグで視点を回せるインタラクティブな表示）に置き換える。
- パノラマの初期視点（heading）は固定または地点ごとにランダムでよい。周囲を見回す操作はユーザーに委ねる。
- 推測用の地図（`GuessMap`）・確定ボタン・スコア表示など、写真パネル方式で使っていた周辺 UI 構造は流用し、置き換えるのはパノラマ表示部分のみとする。
- ストリートビューの読み込みが失敗した場合（パノラマ表示エラー）、既存の `onPhotoFailure`（写真読み込み失敗時の再抽選）と同様に、そのラウンドを再抽選できる仕組みを用意する。

### 4.3 結果画面での表示

- ランダム地点には固有名称・解説文・写真クレジットが存在しない。結果画面（`RoundResultScreen`）は、スポット名や解説を表示する既存の `reveal` ブロックを、ランダム地点でも破綻しない内容に調整する（例: 「この地点」「大阪府内のある地点」といった一般的な文言、または緯度経度の簡易表示）。
- 既存のストリートビュー外部リンク（Google Maps を新規タブで開く機能）は、ランダム地点の正解座標に対しても引き続き提供する。
- 距離・スコアの表示は既存ロジックをそのまま使う。

### 4.4 APIキー未設定時のフォールバック

- ビルド時に Google Maps Platform の API キーが設定されていない場合、アプリは**既存の写真パネル方式（15スポット出題）で動作する**。
- フォールバックの発生をユーザーに気づかれる形で明示する必要はない（既存のゲーム体験がそのまま提供されればよい）が、開発者向けにはビルドログ等で API キー未設定である旨がわかることが望ましい。
- 実行時（地点生成・メタデータ取得・パノラマ表示のいずれか)に回復不能なエラーが発生した場合も、ゲーム全体をクラッシュさせず、既存の `fatalError` フェーズまたはラウンド単位の再抽選で復帰させる。

### 4.5 デザイン要件

- 既存の frontend-design 品質基準（大阪らしさを反映した独自性のあるビジュアル）を継承する。パノラマ表示エリアも既存の `photo-viewer` 相当の枠組み・演出（コーナー装飾、ローディング表現等）と調和させる。
- 日本語 UI。レスポンシブ対応（375px 幅でも操作可能）。
- ストリートビュー・パノラマの読み込み中表示（ローディング状態）を用意する。

## 5. 非機能要件

- 既存のテスト（`npm run test` = `vitest run`）が全てパスし続けること。
- 新規追加ロジック（地点生成、範囲判定、メタデータ確認の呼び出し部分）は、外部通信部分をモック可能な形に分離し、単体テストを追加する。
- API キー未設定でも `npm install && npm run build` が警告なく成功し、アプリがクラッシュしないこと（フォールバック経路の動作を含む）。
- API キー設定時は Cloudflare Pages のビルド時環境変数として注入する。ローカル開発では `.env`（`.env.example` を用意し、`.gitignore` 対象とする）を使う。
- 既存の `SceneProvider` / `Scene` 型やゲームフロー（reducer, phase 遷移）を破壊しない。型を拡張する場合も後方互換を保つ。

## 6. スコープ外

- 大阪府境の厳密なポリゴン判定（矩形判定 + ストリートビュー有無確認で代替する）。
- ランダム地点の難易度推定・分類（`difficulty` は固定値または簡易な扱いでよい）。
- サーバーサイドでの API キー秘匿（プロキシ等）。クライアントから直接 Google Maps Platform を呼び出す前提。
- 写真パネル方式と完全ランダム方式をユーザーが画面上で切り替えられる UI（切替は API キーの有無という設定によってのみ決まる）。
- 大阪府外の地点が稀に出題される可能性を完全に排除すること（矩形判定の性質上、府境近傍で厳密には隣接自治体になるケースは許容する）。

## 7. 受け入れ基準

1. Google Maps API キーが設定された環境で `npm run build` したビルドでは、ラウンド開始時に大阪府内のランダムな地点のストリートビュー・パノラマが表示される。
2. パノラマ表示地点は、ラウンドごとに動的に生成され、既存15スポットの固定リストに限定されない。
3. ストリートビューが取得できない地点を引いた場合、ユーザーに見える形でクラッシュせず、再抽選される。
4. API キーが未設定のビルドでは、既存の写真パネル方式（15スポット出題）でゲームが問題なく遊べる。
5. 結果画面は、ランダム地点出題の場合でもクラッシュせず、距離・スコア・ストリートビュー外部リンクを表示する。
6. `npx vitest run` が全テストパスする。
7. 推測操作（地図へのピン設置、確定、採点ロジック）の挙動に変更がない。
