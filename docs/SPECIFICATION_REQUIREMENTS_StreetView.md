# 要件定義書: GeOsaka ストリートビュー機能

作成日: 2026-07-12 / 作成者: Fable（要件定義担当）

## 1. 背景

GeOsaka は現在、大阪府内スポットの写真を見て場所を推測する「写真パネル方式」のゲームである（[SPECIFICATION_REQUIREMENTS_GeOsaka.md](SPECIFICATION_REQUIREMENTS_GeOsaka.md)）。要件定義書は「将来 Street View に差し替えられるよう SceneProvider を抽象化しておく」ことを既に定めており、`src/scene/types.ts` の `SceneProvider` インターフェースと `Scene.display.kind` がその拡張点として設計済みである。

本書は、そのストリートビュー機能を実際に追加するための要件を定義する。

## 2. 前提・制約の再確認

- 既存要件定義書は **Google Street View API を MVP で不採用**としている（APIキー・課金登録が必須のため）。この方針は本機能でも尊重する。
- プロジェクトは**バックエンド不要・APIキー不要の静的サイト**として構築されている（`docs/SPECIFICATION_DESIGN_CloudflarePages.md`）。APIキーを要する外部サービスを採用する場合は、**未設定でもアプリが壊れないフォールバック**を必須とし、ユーザーに取得手順を案内する。
- 出題（推測対象の写真）を差し替える大がかりな変更ではなく、**既存の写真パネル方式のゲームフローは変更しない**。ストリートビューは追加の探索機能として位置づける。

## 3. 機能要件

### 3.1 配置場所とタイミング

- **ラウンド結果画面（RoundResultScreen）** に、正解地点をストリートビュー（実写の周辺風景）で確認できる機能を追加する。
- 推測前（RoundScreen）には表示しない。理由: 推測前に周辺の実写映像を見せると、写真パネルよりも遥かに多くの手がかり（看板の文字、周辺建物、交通標識等）を与えてしまい、ゲームバランスを崩すため。既存要件の「結果表示後にクレジットを出す」というネタバレ防止方針と一貫させる。
- UI 上は「実際の景色をストリートビューで見る」のような明示的なトグル/ボタンとし、既存の `reveal`（スポット名・解説・クレジット）ブロック付近に配置する。デフォルトで開いた状態にはせず、ユーザー操作で表示する（結果画面の初期表示を軽量に保つため）。

### 3.2 表示内容

- 正解地点（`RoundResult` の答え座標）付近の実写・周辺風景を、**Google Maps の Street View を新規タブで開くリンク**として提供する（技術選定の結果、APIキー・課金登録不要の外部リンク方式を採用。詳細は [SPECIFICATION_TECH_StreetView.md](SPECIFICATION_TECH_StreetView.md)）。
- リンク先で実際に Street View 画像が表示されるか、近傍に画像がない旨が案内されるかは Google Maps 側の挙動に委ね、GeOsaka 側では判定・分岐しない。
- アプリ内にローディング表示や画像取得処理は持たない（外部サイトへの単純な遷移のため）。
- 座標が不正でリンクを生成できない場合も、ラウンド結果画面自体はクラッシュせず、通常の結果表示（距離・スコア・クレジット等）は機能し続ける。

### 3.3 APIキーの扱い

- 技術選定の結果、**APIキー・課金登録・アカウント登録を一切必要としない外部リンク方式**を採用した（Google Maps Embed API 等の埋め込み方式は、利用自体が無料でも Google Cloud 請求先アカウント登録が必須であり、既存要件定義書の「APIキー・課金・アカウント登録不要」原則に反するため不採用）。
- 環境変数・APIキーは一切使用しない。`npm install && npm run build` がクローン直後に追加設定なしで成功する。

### 3.4 デザイン要件

- 既存の frontend-design 品質基準（大阪らしさを反映した独自性のあるビジュアル、生成AIっぽくない配色・タイポグラフィ）を継承する。
- 日本語 UI。レスポンシブ対応（375px 幅でも操作可能）。
- 結果画面の既存レイアウト（`result-hero` / `ResultMap` / `reveal`）と調和する形で追加し、既存要素の視認性を損なわない。

## 4. 非機能要件

- 既存のテスト（`npm run test` = `vitest run`）が全てパスし続けること。
- 新規追加コンポーネントも既存の GuessMap/ResultMap 同様、単体テストでモック可能な形にする。
- APIキー・外部サービス未設定でも `npm run build` が警告なく成功し、アプリがクラッシュしないこと。
- 既存の `SceneProvider` / `Scene` 型やゲームフロー（reducer, phase 遷移）を破壊しない。型を拡張する場合も後方互換を保つ。

## 5. スコープ外

- 推測フェーズ（RoundScreen）でのストリートビュー表示（ゲームバランスを崩すため対象外）。
- 完全な360度パノラマ操作体験の作り込み（採用サービスが提供するビューアの標準機能に留める。自前でパノラマビューアを実装しない）。
- 全15スポットに対する事前の街並み画像の手動キュレーション（技術選定で決めるサービスの動的取得に委ねる）。
- サーバーサイドでのAPIキー秘匿（プロキシ等）。静的サイトのままクライアントから直接呼び出す前提。

## 6. 受け入れ基準

1. ラウンド結果画面のリンクから、正解座標付近の Google Maps Street View を新規タブで開ける。
2. 座標が不正でリンクを生成できない場合も、結果画面はクラッシュせず通常表示を維持する。
3. APIキー・環境変数なしで `npm install && npm run build` がクローン直後に成功する。
4. `npx vitest run` が全テストパスする。
5. 推測フェーズの挙動・見た目に変更がない。
